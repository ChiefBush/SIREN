# SIREN Database Schema

This document describes the Supabase database schema for the SIREN application.

## Tables

### 1. users
Stores user account information and role assignments.

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT UNIQUE NOT NULL,
  full_name TEXT,
  employee_id TEXT UNIQUE,
  role TEXT NOT NULL DEFAULT 'miner' CHECK (role IN ('miner', 'supervisor', 'admin')),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_users_role ON users(role);
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_users_employee_id ON users(employee_id);
```

**Fields:**
- `id`: Unique identifier (UUID)
- `email`: User email address (unique)
- `full_name`: User's full name (letters and spaces only)
- `employee_id`: Employee ID (unique, auto-generated by database trigger)
  - Format: `MIN-0001` (for miners), `SUP-0001` (for supervisors), `ADM-0001` (for admins)
- `role`: User role (miner, supervisor, admin)
- `created_at`: Account creation timestamp
- `updated_at`: Last update timestamp

**Note:** Employee ID is automatically generated when a new user is created. See `DATABASE_TRIGGER.sql` for the trigger setup.

### 2. sensor_data
Stores real-time sensor readings from IoT devices.

```sql
CREATE TABLE sensor_data (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  mq2 NUMERIC,
  mq9 NUMERIC,
  mq135 NUMERIC,
  dht11_temp NUMERIC,
  dht11_humidity NUMERIC,
  timestamp TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  device_id TEXT,
  location TEXT
);

-- Indexes
CREATE INDEX idx_sensor_data_timestamp ON sensor_data(timestamp DESC);
CREATE INDEX idx_sensor_data_device ON sensor_data(device_id);
```

**Fields:**
- `id`: Unique identifier
- `mq2`: MQ2 gas sensor reading
- `mq9`: MQ9 gas sensor reading
- `mq135`: MQ135 gas sensor reading
- `dht11_temp`: DHT11 temperature reading (°C)
- `dht11_humidity`: DHT11 humidity reading (%)
- `timestamp`: Reading timestamp
- `device_id`: IoT device identifier
- `location`: Sensor location

### 3. sensor_alerts
Stores alerts generated from sensor threshold violations.

```sql
CREATE TABLE sensor_alerts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  sensor_type TEXT NOT NULL,
  value NUMERIC NOT NULL,
  threshold NUMERIC NOT NULL,
  status TEXT NOT NULL DEFAULT 'warning' CHECK (status IN ('warning', 'critical', 'acknowledged', 'resolved')),
  message TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  acknowledged_at TIMESTAMP WITH TIME ZONE,
  acknowledged_by UUID REFERENCES users(id),
  created_by UUID REFERENCES users(id)
);

-- Indexes
CREATE INDEX idx_sensor_alerts_status ON sensor_alerts(status);
CREATE INDEX idx_sensor_alerts_created ON sensor_alerts(created_at DESC);
```

**Fields:**
- `id`: Unique identifier
- `sensor_type`: Type of sensor (mq2, mq9, mq135, dht11_temp, dht11_humidity, system)
- `value`: Sensor value that triggered alert
- `threshold`: Threshold value that was exceeded
- `status`: Alert status (warning, critical, acknowledged, resolved)
- `message`: Alert message/description
- `created_at`: Alert creation timestamp
- `acknowledged_at`: When alert was acknowledged
- `acknowledged_by`: User who acknowledged (FK to users)
- `created_by`: User/system that created alert (FK to users)

### 4. attendance
Stores worker attendance records.

```sql
CREATE TABLE attendance (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  date DATE NOT NULL,
  entry_time TIME,
  exit_time TIME,
  hours_worked NUMERIC DEFAULT 0,
  status TEXT DEFAULT 'present' CHECK (status IN ('present', 'absent', 'late', 'early_leave')),
  marked_by UUID REFERENCES users(id),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  UNIQUE(user_id, date)
);

-- Indexes
CREATE INDEX idx_attendance_user ON attendance(user_id);
CREATE INDEX idx_attendance_date ON attendance(date DESC);
```

**Fields:**
- `id`: Unique identifier
- `user_id`: Worker ID (FK to users)
- `date`: Attendance date
- `entry_time`: Entry time
- `exit_time`: Exit time
- `hours_worked`: Calculated hours worked
- `status`: Attendance status
- `marked_by`: User who marked attendance (FK to users)
- `created_at`: Record creation timestamp

### 5. salary_calculations
Stores salary calculation records for workers.

```sql
CREATE TABLE salary_calculations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  period_start DATE NOT NULL,
  period_end DATE NOT NULL,
  hours_worked NUMERIC NOT NULL DEFAULT 0,
  hourly_rate NUMERIC NOT NULL,
  total_amount NUMERIC NOT NULL,
  status TEXT DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'paid')),
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  paid_at TIMESTAMP WITH TIME ZONE
);

-- Indexes
CREATE INDEX idx_salary_user ON salary_calculations(user_id);
CREATE INDEX idx_salary_period ON salary_calculations(period_start, period_end);
```

**Fields:**
- `id`: Unique identifier
- `user_id`: Worker ID (FK to users)
- `period_start`: Pay period start date
- `period_end`: Pay period end date
- `hours_worked`: Total hours worked in period
- `hourly_rate`: Hourly wage rate
- `total_amount`: Calculated total salary
- `status`: Payment status
- `created_at`: Record creation timestamp
- `paid_at`: Payment timestamp

### 6. health_safety_logs
Stores personal health and safety logs entered by workers.

```sql
CREATE TABLE health_safety_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  date DATE NOT NULL,
  fatigue_level TEXT CHECK (fatigue_level IN ('low', 'moderate', 'high', 'severe')),
  symptoms TEXT,
  notes TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_health_user ON health_safety_logs(user_id);
CREATE INDEX idx_health_date ON health_safety_logs(date DESC);
```

**Fields:**
- `id`: Unique identifier
- `user_id`: Worker ID (FK to users)
- `date`: Log date
- `fatigue_level`: Fatigue level (low, moderate, high, severe)
- `symptoms`: Reported symptoms
- `notes`: Additional notes
- `created_at`: Record creation timestamp

### 7. shifts
Stores shift assignments for workers.

```sql
CREATE TABLE shifts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  shift_date DATE NOT NULL,
  start_time TIME NOT NULL,
  end_time TIME NOT NULL,
  notes TEXT,
  assigned_by UUID REFERENCES users(id),
  notified BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- Indexes
CREATE INDEX idx_shifts_user ON shifts(user_id);
CREATE INDEX idx_shifts_date ON shifts(shift_date);
```

**Fields:**
- `id`: Unique identifier
- `user_id`: Worker ID (FK to users)
- `shift_date`: Shift date
- `start_time`: Shift start time
- `end_time`: Shift end time
- `notes`: Additional shift notes
- `assigned_by`: User who assigned shift (FK to users)
- `notified`: Whether worker was notified
- `created_at`: Record creation timestamp

### 8. incidents
Stores incident and hazard reports.

```sql
CREATE TABLE incidents (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  reported_by UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  incident_type TEXT NOT NULL CHECK (incident_type IN ('accident', 'hazard', 'near_miss', 'equipment_failure', 'environmental', 'other')),
  severity TEXT NOT NULL DEFAULT 'low' CHECK (severity IN ('low', 'medium', 'high', 'critical')),
  location TEXT NOT NULL,
  description TEXT NOT NULL,
  date DATE NOT NULL,
  status TEXT DEFAULT 'reported' CHECK (status IN ('reported', 'in_progress', 'resolved', 'closed')),
  reported_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  resolved_at TIMESTAMP WITH TIME ZONE,
  resolved_by UUID REFERENCES users(id)
);

-- Indexes
CREATE INDEX idx_incidents_reported_by ON incidents(reported_by);
CREATE INDEX idx_incidents_status ON incidents(status);
CREATE INDEX idx_incidents_date ON incidents(date DESC);
```

**Fields:**
- `id`: Unique identifier
- `reported_by`: User who reported (FK to users)
- `incident_type`: Type of incident
- `severity`: Incident severity level
- `location`: Incident location
- `description`: Detailed description
- `date`: Incident date
- `status`: Resolution status
- `reported_at`: Report timestamp
- `resolved_at`: Resolution timestamp
- `resolved_by`: User who resolved (FK to users)

## Row Level Security (RLS) Policies

### users
- Users can read their own record
- Admins can read all records
- Supervisors can read miners' records

### sensor_data
- All authenticated users can read sensor data
- Only system/service accounts can insert

### sensor_alerts
- All authenticated users can read alerts
- Admins and supervisors can acknowledge alerts

### attendance
- Users can read their own attendance
- Supervisors and admins can read all attendance
- Supervisors and admins can create/update attendance

### salary_calculations
- Users can read their own salary records
- Admins can read all salary records
- Admins can create/update salary records

### health_safety_logs
- Users can read and create their own logs
- Supervisors and admins can read all logs

### shifts
- Users can read their own shifts
- Supervisors and admins can read all shifts
- Supervisors and admins can create/update shifts

### incidents
- Users can read and create incidents
- Supervisors and admins can read and update all incidents

## Setup Instructions

1. Create a new Supabase project
2. **Run the complete schema file**: `docs/SUPABASE_SCHEMA.sql` in the Supabase SQL Editor
   - This single file contains all tables, indexes, triggers, and RLS policies
   - Employee IDs are automatically generated by the trigger
3. Enable real-time replication in Supabase Dashboard → Database → Replication
   - Enable for: `sensor_data`, `sensor_alerts`, `incidents`

**Note:** For a complete, ready-to-run schema file, use `docs/SUPABASE_SCHEMA.sql` instead of running individual statements from this documentation file.

## Notes

- All timestamps use `TIMESTAMP WITH TIME ZONE` for proper timezone handling
- Foreign key constraints ensure data integrity
- Indexes are created for frequently queried columns
- Unique constraints prevent duplicate records where needed
- Check constraints enforce valid enum values
